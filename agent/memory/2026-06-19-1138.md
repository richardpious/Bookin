assistant: To control tool approval behavior, you need to understand that OpenClaw's security model focuses primarily on **Exec-tool security** (shell commands) and **MCP (Model Context Protocol) tool approval**.

There is no single "requireApproval" global switch for every tool call; instead, it is managed through **security policies** and **ask modes**.

### 1. For Shell Commands (The `exec` tool)
You can fine-tune which shell commands require approval by adjusting the `exec` policy in your `~/.openclaw/openclaw.json`.

*   **`security`**:
    *   `deny`: Blocks all commands (except those on an explicit allowlist).
    *   `allowlist`: Only runs commands on an explicit allowlist.
    *   `full`: Allows all commands (not recommended for production).
*   **`ask`** (this is what you likely want for "interactive" control):
    *   `off`: Never asks.
    *   `on-miss`: Only asks if the command is not on an allowlist or if it's considered risky (the "default" behavior).
    *   `always`: Prompts you for approval for every single shell execution.

**Example configuration:**
```json5
{
  agents: {
    defaults: {
      exec: {
        security: "allowlist", // Force strict checking
        ask: "on-miss"       // Only prompt for unknown/risky commands
      }
    }
  }
}
```

### 2. For External Tools (MCP Tools)
If you are using MCP servers, these are controlled separately by the `defaultToolsApprovalMode` configuration.

*   `auto`: The system attempts to manage approval automatically.
*   `prompt`: Always asks the human for approval.
*   `approve`: Automatically approves all tool calls (High risk, only use for trusted servers).

**Example configuration:**
```json5
{
  tools: {
    mcp: {
      servers: {
        "my-trusted-server": {
          defaultToolsApprovalMode: "approve"
        },
        "my-untrusted-server": {
          defaultToolsApprovalMode: "prompt"
        }
      }
    }
  }
}
```

### How to set this up safely
1.  **Don't set it globally** if you only want it for specific tools. Apply it to the `agents.list[]` section for specific agents, rather than `agents.defaults`.
2.  **Define an Allowlist**: If you switch `security` to `allowlist`, you must provide a list of paths or commands that are permitted to run without a prompt.
3.  **Validate**: Run `openclaw config validate` after making changes. If the config is invalid, the gateway will not boot.

**Recommended Action:**
If you want to reduce prompts, keep `security: "allowlist"` and populate the allowlist with your known, safe commands. Anything outside that list will still trigger an "on-miss" prompt, giving you the balance you're looking for.

user: if i make changes to a plugin hook,do i have to recompile and reregister it in openclaw?
assistant: Yes, if you make changes to a plugin's code, you generally need to restart the Gateway for the changes to take effect.

Here’s the standard breakdown:

*   **Plugin Development (Local):** If you are developing the plugin locally and installed it using `openclaw plugins install --link ./my-plugin`, OpenClaw typically watches that linked directory. However, to ensure the plugin runtime is re-initialized (re-registering its tools, hooks, and services), a Gateway restart is almost always required.
*   **Compilation:** If your plugin is written in a language that requires compilation (like TypeScript or Go) before it can run in OpenClaw, **yes, you must recompile** the plugin after making code changes. The OpenClaw runtime only sees the built/compiled assets defined in your `package.json` (under `openclaw.extensions`) or `plugin.json` manifest.
*   **Registration:** You do **not** need to "reregister" the plugin in the sense of running `openclaw plugins install` again. Once a plugin is installed and configured in your `openclaw.json` (or plugin inventory), it stays registered. Restarting the Gateway (`openclaw gateway restart`) forces the plugin manager to re-read the manifest, re-load the code, and re-register all runtime components.

### Recommended Workflow:
1.  **Modify** your plugin code.
2.  **Recompile** (if using TypeScript, Go, etc.) so the distribution files are updated.
3.  **Restart** the Gateway: `openclaw gateway restart`.
4.  **Verify**: Run `openclaw plugins inspect <plugin-id> --runtime` to confirm the updated version or features are active.

If you have **config reload** enabled, some metadata changes might be hot-applied, but code-level changes in the plugin's entry point almost always require a restart to ensure the new code is loaded into the OpenClaw execution context.
