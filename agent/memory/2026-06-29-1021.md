### How to use it in a hook
In your `after_tool_call` handler you can:

1. **Log or display the input** – `payload.arguments` shows what the user asked the tool to do.
2. **Show the output** – iterate over `payload.result.content` to extract text (or media) and present it to the user.
3. **Handle errors** – check `payload.result.isError` to decide whether to surface an error message.
4. **Correlate with the session** – `payload.sessionKey` lets you tie the call to a specific chat/session if you have multiple ongoing.
5. **Forward to the user** – you can construct a custom message (or attach media) using the data above and send it back through the normal chat channel.

**Example pseudo‑code**

```js
function afterToolCall(payload) {
  const { toolName, arguments, result, sessionKey } = payload;

  // Build a user‑friendly summary
  let msg = `🔧 Tool \`${toolName}\` ran in session ${sessionKey}:\n`;
  msg += `📥 Input: \`${JSON.stringify(arguments)}\`\n`;

  if (result.isError) {
    msg += `❌ Error: ${result.content.map(c => c.text).join(' ')}`;
  } else {
    msg += `📤 Output: ${result.content.map(c => c.text).join(' ')}`;
  }

  // Send `msg` back to the user (via your plugin’s messaging API)
  sendMessageToUser(msg);
}
```

All of the fields listed above are available to you in the `after_tool_call` payload, so you have full visibility into both what was requested and what was returned.

When a plugin registers a `before_tool_call` handler, OpenClaw invokes it **right before** the tool is executed. The payload you get contains everything that was present in the outgoing `toolCall` message (the assistant’s request to run the tool) plus a few bits of session‑level metadata that the system always attaches.

Below is the shape of the object that is passed to your `before_tool_call` callback, based on the format we saw in the session history:

| Field | Type | Description |
|-------|------|-------------|
| `toolCallId` | `string` | The unique identifier that matches the `id` field of the `toolCall` message (e.g. `"call-b6572ce5-c133-4eca-9ab1-e9c2d1fc8801"`). |
| `toolName` | `string` | Name of the tool that is about to be invoked (`"exec"`, `"read"`, `"sessions_history"`, …). |
| `arguments` | `object` | The exact parameter object that will be sent to the tool (the same as `toolCall.arguments`). |
| `partialArgs` *(optional)* | `string` | A string‑ified version of `arguments` – useful for logging or debugging; mirrors the `partialArgs` field you saw in the assistant’s message. |
| `sessionKey` | `string` | Identifier of the session in which the tool call is happening (e.g. `"agent:main:webchat:default"`). |
| `timestamp` | `number` (ms since epoch) | Approximate time when the tool call was queued (matches the `timestamp` field on the assistant’s `toolCall` entry). |
| `__openclaw` | `object` | Internal OpenClaw bookkeeping (`recordTimestampMs`, `seq`, etc.). Usually you can ignore this unless you need low‑level diagnostics. |
| `incomingMetadata` *(optional)* | `object` | The untrusted metadata that arrived with the original user message (the `"sender"` block you showed). It is passed through unchanged so your hook can see who/what triggered the call. |

### How you can use it

- **Log / preview the call** – Show the user what you’re about to run:  
  `About to run \`${payload.toolName}\` with args: ${JSON.stringify(payload.arguments)}`
- **Modify or veto** – If your hook returns a falsy value (or throws), OpenClaw will cancel the tool execution; returning a truthy value lets it proceed.
- **Enrich with session info** – Use `payload.sessionKey` to correlate the call with a specific chat or to store per‑session state.
- **Forward metadata** – The `incomingMetadata` block lets you forward the original sender’s ID or label to logging or analytics systems.

In short, **before the tool runs you have full visibility into the tool’s identity, its arguments, the session it belongs to, and the original incoming metadata**—everything you need to log, transform, or even abort the call.
